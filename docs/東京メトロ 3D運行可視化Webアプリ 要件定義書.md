## 2. 背景

東京メトロの運行状況を、従来の2D路線図や一覧表示ではなく、Webブラウザ上の3Dビューで視覚的に確認できるアプリケーションを構築する。

ODPTの公共交通オープンデータカタログには、東京メトロのGTFS形式の鉄道関連情報と、GTFS-RT形式の鉄道関連リアルタイム情報が掲載されている。GTFS静的情報は路線・駅・時刻表などの基礎データ、GTFS-RTはリアルタイム情報の土台として利用できる。

GTFS-RTはProtocol Buffersベースのデータ形式であり、ブラウザから直接扱うよりも、Ubuntu上の中継APIで取得・デコード・正規化してJSONとして配信する構成が望ましい。GTFS Realtime仕様でも、リアルタイムフィードは既存GTFSフィードに関連付けて扱われる前提になっている。

---

## 3. 目的

本システムの目的は、東京メトロの路線・駅・列車運行状況を3Dで可視化し、運行状態を直感的に把握できるWebアプリを構築することである。

初期段階では、完全な地下空間の再現ではなく、路線ごとに高さを分けた2.5Dネットワークビューを採用する。これにより、見た目の分かりやすさと実装負荷のバランスを取る。

---

## 4. 基本方針

1. フロントエンドはCloudflare Pagesで公開する。
2. APIはUbuntu常時稼働サーバー上で動作させる。
3. API公開にはCloudflare Tunnelを利用し、Ubuntu側の直接ポート公開を避ける。
4. GTFS / GTFS-RTの取得、デコード、データクレンジング、JSON正規化はUbuntu側で行う。
5. フロントエンドはThree.jsで3D表示を行う。
6. 初期版では、東京メトロ全線の完全再現ではなく、段階的に路線を増やす。
7. 運行情報は「参考表示」とし、公式運行判断や乗換案内の代替にはしない。

Cloudflare PagesはカスタムドメインやサブドメインをPagesプロジェクトに割り当てる構成をサポートしており、サブドメインの場合はCNAMEでPages側へ向ける運用が可能である。
Cloudflare Tunnelは、Tunnel作成時に発行される `<UUID>.cfargotunnel.com` へCNAMEを向け、ホスト名からTunnel経由でアプリケーションへルーティングできる。

## 6. スコープ

### 6.1 初期スコープ

| 区分           | 内容                                     |
| ------------ | -------------------------------------- |
| 静的路線表示       | 東京メトロの路線・駅を3D空間に表示する                   |
| 2.5D表示       | 路線ごとに高さを分け、重なりを見やすくする                  |
| 駅ノード表示       | 駅を球体またはラベル付きノードとして表示する                 |
| 路線ライン表示      | 路線を線またはチューブ状オブジェクトで表示する                |
| 列車表示         | GTFS-RTまたは疑似データをもとに列車オブジェクトを表示する       |
| 運行状態表示       | 遅延、運休、アラートを色・アイコン・テキストで表示する            |
| 更新時刻表示       | 最終データ取得時刻、最終画面更新時刻を表示する                |
| API中継        | Ubuntu上でGTFS / GTFS-RTを取得・整形してJSON配信する |
| Cloudflare公開 | フロントはPages、APIはTunnel経由で公開する           |

### 6.2 初期スコープ外

| 区分           | 内容                        |
| ------------ | ------------------------- |
| 完全な地下空間再現    | 実際のトンネル深度、ホーム形状、構内構造の3D再現 |
| 正確な列車GPS位置保証 | GTFS-RTに含まれない位置は補間表示とする   |
| 乗換案内         | 経路検索、運賃計算、所要時間検索          |
| 公式運行情報代替     | 鉄道事業者の公式情報を置き換える用途        |
| ユーザー認証       | 初期版は公開閲覧を前提とし、管理系のみ制限を検討  |
| 多事業者連携       | 初期版は東京メトロを対象とする           |

## 7. システム全体像

```text
[ODPT / GTFS / GTFS-RT]
          ↓
[Ubuntu常時稼働サーバー]
  - データ取得
  - Protocol Buffersデコード
  - データクレンジング
  - JSON正規化
  - API配信
          ↓ Cloudflare Tunnel
[api.<domain>]
          ↓
[Cloudflare Pages]
[viz.<domain>]
          ↓
[ブラウザ]
  - Three.js 3D表示
  - 路線・駅・列車表示
  - 遅延・アラート表示
```

## 9. 非機能要件

### 9.1 可用性

| ID      | 要件                                        |
| ------- | ----------------------------------------- |
| NFR-001 | Ubuntu再起動後にAPIと取得処理が自動復旧する                |
| NFR-002 | GTFS-RT取得失敗時も、前回データまたは静的路線図は表示できる         |
| NFR-003 | Cloudflare Tunnel停止時はAPI不可を画面上で分かりやすく表示する |

---

### 9.2 性能

| ID      | 要件                               |
| ------- | -------------------------------- |
| NFR-010 | 初期表示は通常回線で5秒以内を目標とする             |
| NFR-011 | 3D表示は一般的なPCブラウザで操作可能な軽さを維持する     |
| NFR-012 | 列車位置更新は5〜30秒間隔で設定可能とする           |
| NFR-013 | APIレスポンスは必要最小限に整形し、フロント側の処理量を抑える |

---

### 9.3 セキュリティ

| ID      | 要件                                               |
| ------- | ------------------------------------------------ |
| NFR-020 | ODPT APIキーや認証情報はフロントエンドへ露出させない                   |
| NFR-021 | APIキーは`.env`またはsystemd環境変数で管理する                  |
| NFR-022 | APIはCloudflare Tunnel経由で公開し、Ubuntuの直接公開ポートを最小化する |
| NFR-023 | 管理用エンドポイントはローカル限定またはCloudflare Access等で保護する      |
| NFR-024 | 利用条件、出典、免責事項を画面上に表示する                            |

---

### 9.4 保守性

| ID      | 要件                                     |
| ------- | -------------------------------------- |
| NFR-030 | フロントエンド、バックエンド、データ取得処理を分離する            |
| NFR-031 | 路線定義、色、表示高さは設定ファイルで管理する                |
| NFR-032 | データ取得ロジックと3D描画ロジックを密結合させない             |
| NFR-033 | Claude Codeで修正しやすいディレクトリ構成とREADMEを整備する |

---

### 9.5 法務・利用条件

| ID      | 要件                         |
| ------- | -------------------------- |
| NFR-040 | ODPTの利用条件を確認し、遵守する         |
| NFR-041 | データ出典を画面およびREADMEに明記する     |
| NFR-042 | 「本アプリは非公式の可視化PoCである」旨を明記する |
| NFR-043 | リアルタイム情報の正確性を保証しない免責文を表示する |

ODPTでは、開発者サイトへのユーザー登録後、API仕様に沿って公共交通データを利用でき、利用条件の遵守が求められている。

## 11. データ要件

### 11.1 静的データ

| データ        | 内容           |
| ---------- | ------------ |
| routes     | 路線ID、路線名、路線色 |
| stops      | 駅ID、駅名、緯度、経度 |
| trips      | 運行ID、路線ID、方向 |
| stop_times | 停車順、駅、時刻     |
| shapes     | 路線形状座標       |

---

### 11.2 動的データ

| データ         | 内容                |
| ----------- | ----------------- |
| vehicle     | 列車・車両位置情報         |
| trip_update | 運行・遅延・到着予測情報      |
| alert       | 運休、遅延、注意情報        |
| feed_header | GTFS-RT更新時刻、バージョン |

実際にどのフィード項目が利用可能かは、ODPT開発者サイトのAPI仕様および東京メトロ提供データの内容を確認して確定する。

## 14. 主なリスク

| リスク    | 内容                         | 対応                            |
